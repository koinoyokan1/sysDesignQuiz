How websocket works?
$$$$$
In a distributed websocket architecture, there are multiple websocket gateway servers, each maintaining a subset of active websocket connections. When a client connects through the load balancer, it gets attached to one websocket gateway, which keeps the long-lived TCP/websocket connection open. The system stores a mapping like (userId, deviceId) -> websocketGateway in a Presence KV store such as Redis. When a message needs to be delivered, the Fanout Service first looks up which websocket gateways currently own the recipient’s active connections, then routes the message to those gateways. The gateways finally push the message over their existing websocket connections to the client devices. This design allows websocket handling to scale horizontally across many gateway servers while still supporting realtime low-latency delivery and multi-device fanout.
Note that the user,device: server mapping is only needed to fan-out msg to diff WS. If it was a single WS conn where a user uses a WB to send a msg, then LB already has the info and routes to the right WS gateway.

How E2EE works in group chat that involves blob store for attachments
$$$$$
When a user sends a media attachment in an end-to-end encrypted chat system, the client first generates a random symmetric mediaKey and uses it to encrypt the attachment locally before uploading it to blob storage. Since the blob store only sees encrypted bytes and does not possess the mediaKey, it cannot access the actual media contents. The blob store URL is included as part of the encrypted message metadata.
For the actual chat message, the sender generates another symmetric key called msgKey and encrypts the message payload using it. The payload may include text, media metadata, blob URLs, and the mediaKey itself. In group chats, the system does not encrypt the entire message separately using every participant’s public key because that would be computationally expensive. Instead, the msgKey is encrypted separately for each recipient device using that device’s public key and attached to the message. Each recipient uses their private key to decrypt the msgKey, then uses the msgKey to decrypt the message payload, retrieve the mediaKey and blob URL, download the encrypted media from blob storage, and finally decrypt the media locally using the mediaKey.

How to solve hot partitioning when storing followerId, followeeId table in insta? $$$$$ In distributed databases like Apache Cassandra or Amazon DynamoDB, all rows with the same partition key are stored together, so if we model followers as PK = followeeId, then a celebrity with hundreds of millions of followers causes all reads and writes to hit the same partition, creating a hot partition that overloads a few nodes and slows the system. To solve this, we shard the celebrity’s followers across multiple partitions using a bucketed key like PK = followeeId_bucket, where bucket = hash(followerId) % N; this spreads followers evenly across many partitions and machines, allowing follower reads and writes to scale horizontally while preventing any single partition from becoming too large or too hot.
insta feed gen algo $$$$$ At internet scale, Instagram uses a hybrid fanout strategy for feed generation. When a non-celebrity posts, their content is fanned out asynchronously via a message queue (like Kafka) to fanout workers, which write the post ID and its creation timestamp as the score into each follower's pre-built feed list — stored in a distributed Redis cluster, sharded by user ID, so each user's feed sorted set lives close to their region. When a celebrity posts, the fanout is skipped entirely; instead their posts sit in their own per-celebrity sorted list, also in Redis. At read time, the API server (not a separate store) instantiates a min-heap in memory — ephemeral, living only for the duration of that request — and performs a k-way merge between the requesting user's pre-built feed sorted set and one sorted list per celebrity they follow, always popping the highest timestamp across all lists to produce a globally ordered reverse-chronological feed. The heap is never persisted; it's reconstructed per request on the API server handling that user's feed call, which is why the API tier needs to be stateless and horizontally scalable. The pre-built feed in Redis is capped (say at 800 posts) and trimmed on write to bound memory, and the Redis cluster itself is configured with replication and persistence so feed data survives node failures.Sonnet 4.6
How a user always hits their closest server? $$$$$ Instagram uses a global Anycast DNS and CDN layer (Cloudflare or their own PoPs) where the same domain instagram.com resolves to different IP addresses depending on where the DNS query originates. A user in Mumbai gets routed to the Mumbai PoP, a user in London hits a European PoP. This is handled entirely at the DNS and BGP routing level before the request even reaches Instagram's infrastructure. The closest Point of Presence (PoP) terminates the connection, handles static assets, and forwards API requests to the nearest application server cluster. The issue here is that "nearest" by network hops isn't always nearest by latency — sometimes a geographically farther server has better peering and wins. This is resolved by using latency-based routing (Route 53's latency routing for example) rather than pure geographic routing, so the decision is based on measured round-trip time, not physical distance.
How a user's cache is kept close to them? $$$$$ Each region runs an independent Redis cluster, and at account creation the user is assigned a home region. All feed writes (fanout) and feed reads go to that region's Redis cluster. The routing service reads the user's home region from a lightweight global directory (a fast key-value store like Zookeeper or a replicated config service) and forwards the request to the correct regional Redis. The problem is cross-region followers — if your followers are in Tokyo and London, fanout workers in your home region must write into foreign regional Redis clusters over the WAN, adding latency to every write. This is resolved by making fanout asynchronous via a message queue like Kafka, so the post write returns immediately to the user and fanout propagates in the background. Staleness in foreign caches of a few seconds is accepted as eventual consistency.
How a user's DB data is kept close to them? $$$$$ At account creation, the user's ID is generated with an embedded region identifier, so their row always hashes to a shard that is physically pinned to their home region's database cluster. A global shard map (maintained in a config service) explicitly maps shard ranges to regions, so shard placement is never random. Each region runs a primary DB for its own shards and maintains read replicas of foreign shards for cross-region reads. The hard problem is relational data — your followers live in 50 different regional shards and any social graph query inherently crosses regions. How multipart upload works $$$$$ A typical multipart upload REST flow starts with the client calling something like POST /uploads/initiate to create an upload session, and the server returns an uploadId plus optionally presigned URLs for chunk uploads. The client then splits the file into parts and uploads each independently using calls like PUT /uploads/{uploadId}/parts/{partNumber}, often in parallel and directly to object storage. The server tracks metadata for uploaded parts such as part number, checksum/ETag, and status. If the upload is interrupted, the client can call GET /uploads/{uploadId}/status to discover which parts already succeeded and resume only missing chunks. Once all parts are uploaded, the client finalizes the upload using POST /uploads/{uploadId}/complete with the list of uploaded parts and their ETags, after which the storage system validates and assembles the final object. Optionally, DELETE /uploads/{uploadId} aborts and cleans up incomplete uploads.

How kafka partition incrase works?
$$$$$
Three fixes:
1. Plan partition count upfront:
Start with 100 partitions even at low volume. Empty partitions are cheap. Rebalancing is expensive. Never need to add partitions.
2. Use consistent hashing for partition assignment:
Instead of hash(key) % N — use a consistent hash ring. Adding partitions only remaps a fraction of keys, not all of them. Minimal disruption.
3. Reprocessing after repartition:
If you must add partitions — replay all in-flight events from the beginning of the affected time window. Expensive but correct.
