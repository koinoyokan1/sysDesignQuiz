How websocket works?
$$$$$
In a distributed websocket architecture, there are multiple websocket gateway servers, each maintaining a subset of active websocket connections. When a client connects through the load balancer, it gets attached to one websocket gateway, which keeps the long-lived TCP/websocket connection open. The system stores a mapping like (userId, deviceId) -> websocketGateway in a Presence KV store such as Redis. When a message needs to be delivered, the Fanout Service first looks up which websocket gateways currently own the recipient’s active connections, then routes the message to those gateways. The gateways finally push the message over their existing websocket connections to the client devices. This design allows websocket handling to scale horizontally across many gateway servers while still supporting realtime low-latency delivery and multi-device fanout.
Note that the user,device: server mapping is only needed to fan-out msg to diff WS. If it was a single WS conn where a user uses a WB to send a msg, then LB already has the info and routes to the right WS gateway.

How E2EE works in group chat that involves blob store for attachments
$$$$$
When a user sends a media attachment in an end-to-end encrypted chat system, the client first generates a random symmetric mediaKey and uses it to encrypt the attachment locally before uploading it to blob storage. Since the blob store only sees encrypted bytes and does not possess the mediaKey, it cannot access the actual media contents. The blob store URL is included as part of the encrypted message metadata.
For the actual chat message, the sender generates another symmetric key called msgKey and encrypts the message payload using it. The payload may include text, media metadata, blob URLs, and the mediaKey itself. In group chats, the system does not encrypt the entire message separately using every participant’s public key because that would be computationally expensive. Instead, the msgKey is encrypted separately for each recipient device using that device’s public key and attached to the message. Each recipient uses their private key to decrypt the msgKey, then uses the msgKey to decrypt the message payload, retrieve the mediaKey and blob URL, download the encrypted media from blob storage, and finally decrypt the media locally using the mediaKey.
